# IAP/OTA Bootloader 上位机通信协议指南

本文档面向上位机开发者，用于和本项目的 STM32F407 IAP/OTA Bootloader 联调。只要按照本文档实现，上位机即可通过 UART 或 Ethernet TCP 与设备完成协议交互、参数读取和固件升级。

## 1. 协议概述

本项目采用统一的二进制协议帧，UART 和 Ethernet TCP 复用同一套帧格式与命令定义。

当前支持的传输方式：

- UART：串口协议收发
- Ethernet TCP：RAW API TCP Server，默认监听端口 `5000`

协议特点：

- 小端序
- 固定帧头 `0xAA55`
- 支持普通命令、升级命令、参数读写命令
- 支持 `ACK / NACK / BUSY`
- TCP 侧支持粘包、拆包、随机分片
- 升级通道互斥：同一时刻只允许一个传输通道占用升级会话

## 2. 传输层约定

### 2.1 UART

- 波特率：`115200`
- 数据位：`8`
- 停止位：`1`
- 校验位：`None`

### 2.2 Ethernet TCP

- 监听端口：`5000`
- 连接模式：单连接
- 一次只允许一个客户端连接
- TCP 是字节流，不能假定“一次 recv 就是一帧”

### 2.3 通道互斥

升级期间，一个通道先发 `CMD_UPDATE_START` 后，后续 `DATA / END` 必须继续走同一通道。

如果另一个通道试图抢占升级会话，设备会返回 `CMD_BUSY`。

## 3. 协议帧格式

所有命令和响应都使用同一帧格式：

| 字段 | 偏移 | 长度 | 说明 |
|---|---:|---:|---|
| Header | 0 | 2 | 固定帧头 `0xAA55` |
| Cmd | 2 | 2 | 命令字 |
| Length | 4 | 2 | Data 字段长度，单位字节 |
| Data | 6 | N | 负载数据，长度由 Length 指定 |
| Reserved | 6+N | 4 | 保留字段，当前固定填 `0x00000000` |
| CRC16 | 10+N | 2 | CRC16 校验值 |

### 3.1 编码规则

- 所有多字节字段均为小端序
- `Length` 表示 `Data` 的字节数，不包含头、保留区和 CRC
- `Reserved` 当前发送端填 `0x00000000`
- `CRC16` 覆盖从 `Header` 到 `Reserved` 的全部字节，不包含最后 2 字节 CRC 本身

### 3.2 最小帧长度

最小帧长度为 `12` 字节：

- Header 2
- Cmd 2
- Length 2
- Reserved 4
- CRC16 2

当 `Length = 0` 时，整帧就是 12 字节。

### 3.3 最大 Data 长度

`PROTOCOL_MAX_DATA_LEN = 256`

也就是说，`Length` 最大不能超过 256。

## 4. CRC16 规则

- CRC16 由设备在封包时自动计算
- 上位机必须按同样规则计算
- 校验失败时设备会拒绝该帧

计算范围：

- 从 `Header` 开始
- 到 `Reserved` 结束
- 不包含 `CRC16` 字段本身

## 5. 命令定义

### 5.1 系统命令

| 命令 | 值 | 说明 |
|---|---:|---|
| `CMD_PING` | `0x0100` | 心跳 / 回显测试 |
| `CMD_RESET` | `0x0101` | 软件复位 |
| `CMD_GET_VERSION` | `0x0102` | 获取 Boot 版本信息 |

### 5.2 参数命令

| 命令 | 值 | 说明 |
|---|---:|---|
| `CMD_PARAM_READ` | `0x0500` | 读取参数区摘要 |
| `CMD_PARAM_WRITE` | `0x0501` | 写入参数（当前实现为计数/保存） |

### 5.3 升级命令

| 命令 | 值 | 说明 |
|---|---:|---|
| `CMD_UPDATE_START` | `0x0300` | 开始升级 |
| `CMD_UPDATE_DATA` | `0x0301` | 发送升级数据 |
| `CMD_UPDATE_END` | `0x0302` | 结束升级并校验 |
| `CMD_UPDATE_ABORT` | `0x0303` | 中止升级（当前未在协议处理器中实现，预留） |

### 5.4 响应命令

| 命令 | 值 | 说明 |
|---|---:|---|
| `CMD_ACK` | `0xFF00` | 成功应答 |
| `CMD_NACK` | `0xFF01` | 失败应答 |
| `CMD_BUSY` | `0xFF02` | 设备忙 / 会话被其他通道占用 |

## 6. 响应规则

不同命令的返回帧命令字不完全一致，需按下面规则处理：

- `CMD_PING`：成功时回 `CMD_PING`，数据原样回显
- `CMD_GET_VERSION`：成功时回 `CMD_ACK`
- `CMD_PARAM_READ`：成功时回 `CMD_ACK`
- `CMD_PARAM_WRITE`：成功时回 `CMD_ACK`
- `CMD_RESET`：成功时回 `CMD_ACK`
- `CMD_UPDATE_START / DATA / END`：成功时回 `CMD_ACK`
- 其他未知命令：回 `CMD_NACK`
- 升级期间被其他通道抢占：回 `CMD_BUSY`

上位机应同时检查：

- 返回帧的 `Cmd`
- `Length`
- `Data` 内容
- `CRC16`

## 7. 数据结构定义

### 7.1 版本信息返回结构

`CMD_GET_VERSION` 的数据区返回如下结构：

```c
typedef struct {
	 uint32_t boot_version;
	 uint32_t app_version;
	 uint32_t hw_version;
	 char     build_date[12];
} VersionInfo_t;
```

字段含义：

- `boot_version`：Bootloader 版本
- `app_version`：应用版本（当前示例中为编译时写死值）
- `hw_version`：硬件版本
- `build_date`：编译日期字符串，11 字符 + 结尾 `\0`

### 7.2 参数摘要返回结构

`CMD_PARAM_READ` 的数据区返回如下结构：

```c
typedef struct {
	 uint32_t boot_version;
	 uint32_t boot_run_count;
	 uint32_t run_app_version;
	 uint8_t  app_a_status;
	 uint8_t  app_b_status;
} ParamSummary_t;
```

字段含义：

- `boot_version`：Bootloader 版本
- `boot_run_count`：Boot 启动/运行计数
- `run_app_version`：当前 Run 区版本
- `app_a_status`：A 区状态
- `app_b_status`：B 区状态

状态值来自固件参数区定义：

- `0`：`APP_STATUS_INVALID`
- `1`：`APP_STATUS_VALID`
- `2`：`APP_STATUS_UPDATING`
- `3`：`APP_STATUS_ERROR`

### 7.3 升级开始结构

`CMD_UPDATE_START` 的数据区必须为：

```c
typedef struct {
	 uint32_t target;
	 uint32_t version;
	 uint32_t size;
	 uint32_t crc32;
} __attribute__((packed)) UpdateStartInfo_t;
```

字段含义：

- `target`：目标升级区
  - `1` = `APP_AREA_A`
  - `2` = `APP_AREA_B`
- `version`：目标固件版本
- `size`：固件总长度，单位字节
- `crc32`：固件完整 CRC32，用于 `CMD_UPDATE_END` 最终校验

### 7.4 升级数据结构

`CMD_UPDATE_DATA` 的数据区就是固件分片原始字节流，没有额外头部。

要求：

- `Length > 0`
- `Length % 4 == 0`
- 所有分片累加长度不能超过 `CMD_UPDATE_START` 中声明的 `size`

### 7.5 升级结束结构

`CMD_UPDATE_END` 当前不要求额外数据区，通常发送 `Length = 0` 的空帧即可。

## 8. 升级流程

推荐上位机按以下顺序实现：

1. 建立连接 / 打开串口
2. 可选：`CMD_PING`
3. 可选：`CMD_GET_VERSION`
4. 发送 `CMD_UPDATE_START`
5. 等待 `CMD_ACK`
6. 按分片发送若干 `CMD_UPDATE_DATA`
7. 每包等待 `CMD_ACK`
8. 发送 `CMD_UPDATE_END`
9. 等待 `CMD_ACK`
10. 如需要，发送 `CMD_RESET`

### 8.1 Start

`CMD_UPDATE_START` 进入升级会话时，设备会：

- 检查目标区是否合法
- 检查 size 是否合理
- 擦除目标 Flash 区域
- 在参数区记录更新状态
- 占用升级会话锁

若另一个通道正在占用升级会话，设备返回 `CMD_BUSY`。

### 8.2 Data

`CMD_UPDATE_DATA` 发送过程中，设备会：

- 检查是否为当前 owner 通道
- 检查是否处于升级中
- 检查长度是否为 4 字节对齐
- 检查是否越界
- 写入目标 Flash

若当前通道不是 owner，设备返回 `CMD_BUSY`。

### 8.3 End

`CMD_UPDATE_END` 发送后，设备会：

- 检查是否为当前 owner 通道
- 检查接收总长度是否等于 Start 声明长度
- 计算目标区 CRC32
- 与 Start 中的 `crc32` 比对
- 成功后更新参数区状态并释放会话锁

若 CRC 不匹配，设备返回 `CMD_NACK`，并标记目标区无效。

## 9. BUSY / NACK 的区别

上位机应明确区分两种失败：

- `CMD_BUSY`
  - 表示设备当前被其他通道占用
  - 适合重试

- `CMD_NACK`
  - 表示参数错误、长度错误、CRC 错误、Flash 失败等
  - 不建议盲目重试，应该先修正请求内容

建议上位机逻辑：

- 收到 `CMD_BUSY`：延迟后重试或提示“通道被占用”
- 收到 `CMD_NACK`：记录错误码并终止当前会话

## 10. 分片与对齐要求

### 10.1 TCP 粘包 / 拆包

TCP 上位机必须按字节流思维实现：

- 允许一帧拆成多个 TCP 包发送
- 允许多个协议帧在一个 TCP 包中连续发送
- 设备内部已经处理粘包和拆包

### 10.2 升级数据对齐

升级数据 `CMD_UPDATE_DATA` 必须满足：

- 每包长度为 4 字节对齐
- 总长度也应为 4 字节对齐

否则设备会返回 `CMD_NACK`。

## 11. 典型报文示例

以下示例使用十六进制描述，字段均为小端序。

### 11.1 PING

请求：

```text
55 AA 00 01 04 00 31 32 33 34 00 00 00 00 xx xx
```

说明：

- `Cmd = 0x0100`
- `Length = 4`
- `Data = 31 32 33 34`

响应：

```text
55 AA 00 01 04 00 31 32 33 34 00 00 00 00 xx xx
```

说明：

- 响应命令仍然是 `CMD_PING`
- Data 原样回显

### 11.2 GET_VERSION

请求：

```text
55 AA 02 01 00 00 00 00 00 00 xx xx
```

响应：`CMD_ACK`，Data 为 `VersionInfo_t`

### 11.3 PARAM_READ

请求：

```text
55 AA 00 05 00 00 00 00 00 00 xx xx
```

响应：`CMD_ACK`，Data 为 `ParamSummary_t`

### 11.4 UPDATE_START

请求数据区结构：

```text
target   = 0x00000001 or 0x00000002
version  = 固件版本
size     = 固件总字节数
crc32    = 固件 CRC32
```

请求示例（只示意 Data 部分）：

```text
[target 4B][version 4B][size 4B][crc32 4B]
```

响应：`CMD_ACK` / `CMD_BUSY` / `CMD_NACK`

### 11.5 UPDATE_DATA

请求：

```text
[原始固件分片字节流]
```

响应：`CMD_ACK` / `CMD_BUSY` / `CMD_NACK`

### 11.6 UPDATE_END

请求：空帧或 `Length = 0`

响应：`CMD_ACK` / `CMD_BUSY` / `CMD_NACK`

### 11.7 RESET

请求：空帧或 `Length = 0`

响应：`CMD_ACK`，随后设备复位

## 12. 上位机实现建议

### 12.1 架构建议

建议上位机内部拆成 4 层：

1. 物理层
	- 串口 / TCP Socket
2. 帧层
	- 按本文档组帧、拆帧、CRC16 校验
3. 命令层
	- Ping / Version / Param / Update
4. 会话层
	- 处理升级流程、Busy 重试、超时、异常恢复

### 12.2 必须实现的能力

上位机至少要支持：

- 小端序编码/解码
- CRC16 计算
- TCP 粘包/拆包处理
- ACK / NACK / BUSY 状态判断
- 升级分片发送
- 升级完成后的 Reset

### 12.3 建议的超时设置

建议上位机至少设置：

- `PING / GET_VERSION / PARAM_READ`：较短超时
- `UPDATE_START`：较长超时
- `UPDATE_DATA`：中等超时，允许连续分片发送
- `UPDATE_END`：较长超时，因设备可能需要做 CRC 校验和参数保存

## 13. 联调验收清单

上位机开发完成后，建议按下面清单验收：

1. `CMD_PING` 正常回显
2. `CMD_GET_VERSION` 返回版本结构正确
3. `CMD_PARAM_READ` 返回参数结构正确
4. `CMD_UPDATE_START` 正常进入升级
5. `CMD_UPDATE_DATA` 连续发送无丢包
6. `CMD_UPDATE_END` CRC 正确时返回 `CMD_ACK`
7. 错误数据长度返回 `CMD_NACK`
8. 两个通道同时抢升级时，非 owner 通道返回 `CMD_BUSY`
9. TCP 断开后再次连接，协议状态恢复正常

## 14. 当前协议约束汇总

- 帧头固定：`0xAA55`
- 最大负载：`256` 字节
- 数据区小端序
- 传输层支持 UART 和 TCP
- TCP 默认监听端口 `5000`
- 升级数据单包长度要求 4 字节对齐
- 升级期间只允许一个通道占用会话
- 非 owner 升级请求返回 `CMD_BUSY`

## 15. 推荐的上位机开发顺序

1. 先实现帧编解码和 CRC16
2. 再实现 `PING / GET_VERSION / PARAM_READ`
3. 再实现 TCP 粘包/拆包
4. 再实现 `UPDATE_START / DATA / END`
5. 最后加 `RESET`、Busy 重试和断线重连

这样可以最快完成联调闭环。
